3. Phase 1: Critical Fixes & Foundation

Goal: Stabilize the existing pipeline to ensure reliable, end-to-end processing and secure report delivery. This phase addresses the most severe issues preventing the system from functioning correctly.

Task 1.1: Implement Server-Side Analysis Trigger

Problem: The current client-side trigger is unreliable and fails silently if the user navigates away.
Solution: Use a Supabase Database Webhook to trigger the analysis from the backend upon a new policy insertion.

Implementation Steps:

  1. Create a New API Endpoint in the CRM (Vercel):
    • File: src/app/api/policies/auto-analyze/route.ts
    • Method: POST
    • Logic: This endpoint will receive the webhook from Supabase. It should validate the secret, parse the payload, and then dispatch the analysis job to the Railway API as the current /api/policies/[id]/analyze endpoint does. This centralizes the dispatch logic.
  2. Create the Database Trigger Function:
    • File: supabase/migrations/YYYYMMDDHHMMSS_add_auto_analyze_trigger.sql
    • SQL: Use the plpgsql function provided in the architecture document to perform an http_post to the new Vercel endpoint. Ensure the WEBHOOK_SECRET is stored securely in Supabase secrets.
  3. Create the Database Trigger:
    • File: supabase/migrations/YYYYMMDDHHMMSS_add_auto_analyze_trigger.sql
    • SQL: Create an AFTER INSERT trigger on the insurance_policies table that executes the new function for each row.
  4. Deprecate Client-Side Trigger: Remove the automatic POST request from the insurance-policies-section.tsx component.

Task 1.2: Fix Report Delivery & Storage

Problem: Reports are generated but not saved or delivered to the user. This involves multiple related gaps (Gaps #1, #2, #4, #5).
Solution: Configure the Railway API to upload the generated PDF report to a new Supabase Storage bucket and update the CRM to allow users to download it.

Implementation Steps:

  1. Create New reports Bucket in Supabase Storage: Use the Supabase dashboard to create a new private bucket named reports.
  2. Add report_storage_path Column to Database:
    • File: supabase/migrations/YYYYMMDDHHMMSS_add_report_storage_path.sql
    • SQL: ALTER TABLE insurance_policies ADD COLUMN report_storage_path TEXT;
  3. Configure Supabase Client in Railway API:
    • Add SUPABASE_URL and SUPABASE_KEY (the service_role key) to Railway environment variables.
    • Install the supabase-py library: pip install supabase
    • In orchestrator.py, after the report is generated, add a step to upload the file to the reports bucket and get its path.
  4. Update Callback Payload: In orchestrator.py, add the report_storage_path to the JSON payload sent back to the Vercel CRM webhook.
  5. Update CRM Webhook Handler:
    • File: /api/webhook/analysis-complete/route.ts
    • Modify the handler to receive report_storage_path and save it to the corresponding insurance_policies record.
  6. Add Download Button to CRM UI:
    • File: insurance-policies-section.tsx
    • When a policy has a completed status and a report_storage_path, display a "Download Report" button. The button should generate a signed URL for the report_storage_path and open it in a new tab.

Task 1.3: Enforce Webhook Signature Verification

Problem: Webhooks are not verified, creating a security vulnerability (Gap #3).
Solution: Enforce HMAC-SHA256 signature verification on all incoming webhooks.

Implementation Steps:

  1. Railway API: In webhook.py, uncomment or implement the verify_signature dependency for the /webhook/policy-uploaded endpoint.
  2. Vercel CRM: In /api/webhook/analysis-complete/route.ts, implement the same HMAC verification logic using the Node.js crypto module.
  3. Secret Management: Ensure the shared secret is stored securely in both Vercel and Railway environment variables and is not hardcoded.

Task 1.4: Set Up Basic Monitoring and Logging

Problem: No visibility into errors or system performance.
Solution: Integrate Sentry for error tracking and implement basic structured logging.

Implementation Steps:

  1. Integrate Sentry:
    • Add the Sentry SDK to both the Vercel (TypeScript) and Railway (Python) projects.
    • Initialize Sentry with the appropriate DSN in the application startup.
    • Wrap main processing logic in try...except blocks that capture exceptions to Sentry.
  2. Implement Structured Logging:
    • Use structlog in the Railway API to log key events (e.g., analysis_started, extraction_failed, analysis_completed) with context like policy_id and tenant_id.

4. Phase 2: Quality & Feature Enhancement

Goal: Dramatically improve the quality of data extraction and the analytical depth of the reports. Introduce key user-facing features like real-time progress updates.

Task 2.1: Integrate Advanced PDF Extraction

Problem: pdfplumber is not sufficient for scanned documents or complex layouts.
Solution: Integrate the Unstructured.io API as the primary extraction strategy.

Implementation Steps:

  1. Create EnhancedPDFExtractor Class: In the Railway API, create a new class as outlined in the architecture document.
  2. Implement Multi-Strategy Logic: Implement the extract_from_url method that tries Unstructured.io first and falls back to pdfplumber.
  3. Handle Unstructured.io Output: Adapt the downstream code to handle the richer, element-based output from Unstructured.io, particularly for tables and titles.
  4. API Key Management: Securely store the Unstructured.io API key in Railway environment variables.

Task 2.2: Implement Structured Data Extraction

Problem: Key policy data (limits, dates, etc.) is buried in unstructured text and not used in the analysis.
Solution: Create a pre-AI regex-based extraction layer and a new database table to store this structured data.

Implementation Steps:

  1. Create policy_structured_data Table: Apply the SQL migration to add the new table.
  2. Develop StructuredDataExtractor Class: In the Railway API, build the Python class with regex patterns for common policy fields.
  3. Integrate into Orchestrator: In orchestrator.py, after text extraction, run the StructuredDataExtractor and save the results to the new table.
  4. Enhance AI Prompts: Modify the claude_analyzer.py to fetch this structured data and inject it into the prompt for Phase 2 analysis, providing the AI with critical context.

Task 2.3: Implement Real-Time Progress Updates

Problem: The UI provides no feedback during the 1-3 minute analysis time.
Solution: Use Supabase Realtime to push status updates from the backend to the frontend.

Implementation Steps:

  1. Enable Realtime on insurance_policies Table: In the Supabase dashboard, enable Realtime for the table.
  2. Update Status in Railway API: Throughout orchestrator.py, update the analysis_status in the database at each major step (e.g., extracting_data, analyzing_policy, generating_report).
